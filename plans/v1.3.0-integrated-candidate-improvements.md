## 1. No pruning mandate — the loop only ever adds

**Problem.** `claude-md-tidy-reflect` Step 2 mines evidence for signals to *add* (new tests, steps, verdicts, caveats). Nothing in the loop ever asks "which existing instruction proved useless, wrong, or harmful?" Every reflection cycle grows `claude-md-tidy/SKILL.md`; nothing shrinks it.

**Why it matters.** A skill whose entire purpose is fighting instruction bloat will, over repeated self-improvement cycles, become exactly the kind of overgrown document it exists to fix. This is the most structurally ironic gap in the suite.

**Proposed fix.** Add a signal to `claude-md-tidy-reflect` Step 2 (Extract lessons): *"Instruction that produced no value across N processed run records — never triggered, never cited as the reason a CHALLENGE/DELETE/RELOCATE fired — is a pruning candidate."* Add a corresponding step in the Apply phase that removes it (with evidence: which records were checked and found the instruction inert) rather than only adding.

**Competitive cross-check.** See item 11 below — `tsalkin/claude-memory-hygiene` surfaces the same "the tool that fights bloat accumulates its own bloat" irony one level down, in the suite's `RUNS.md` data file rather than its `SKILL.md` methodology. The two fixes are complementary, not overlapping: this item prunes *rules*, item 11 prunes *records*.

**Files:** `skills/claude-md-tidy-reflect/SKILL.md` (Step 2, Step 5).
**Bump:** MINOR (new signal + apply step).

## 2. Unbounded, non-claim-driven repo survey

**Problem.** `claude-md-tidy` Step 2 says to survey "full file tree; README and everything under docs/; every skill/command; settings.json; manifests; CI; recent git history" with no proportionality to the size of the CLAUDE.md being tidied or the size of the repo. The comprehension gate's exit condition ("if you can't [answer], keep reading") has no upper bound.

**Why it matters.** For a 60-line CLAUDE.md in a large monorepo, a full survey can cost far more than the tidy is worth, with no signal to stop early. A skill meant to save attention shouldn't spend unbounded attention to run.

**Proposed fix.** Make the survey **claim-driven**: extract every concrete claim the CLAUDE.md makes (file paths, command names, workflow descriptions, tool references) first, then verify only those, plus a fixed, cheap general-orientation pass (top-level README, `.claude/skills`+`.claude/commands` listing, `settings.json` — no deep read unless a claim points there). Keep the comprehension gate but redefine "keep reading" as "keep resolving unverified claims," which is inherently bounded by the CLAUDE.md's own length.

**Competitive cross-check.** `severity1/claude-code-auto-memory` runs its background maintenance in an **isolated sub-agent** specifically so the main session's context/token budget is untouched. Independent of making the survey claim-driven, delegating the survey (or the reflect skill's evidence-gathering pass) to a forked sub-agent is a second, complementary way to bound its cost — worth considering as an implementation detail once the claim-driven scope change lands, not a substitute for it.

**Files:** `skills/claude-md-tidy/SKILL.md` (Step 2).
**Bump:** MINOR (changes survey scope/cost, not the workflow contract).

## 3. Single-run lessons become permanent rules on n=1 evidence

**Problem.** `claude-md-tidy-reflect` folds a lesson into the suite from a single overridden verdict in a single repo. The "verify the tag" check in Step 2 is the same model re-checking its own fresh judgment — weak independence. There is no mechanism to later demote a lesson that turns out to be wrong or repo-specific after all.

**Why it matters.** At realistic run volume, demanding 2+ occurrences before acting would mean the loop rarely learns anything — but treating every first sample as generalized truth risks baking overfit or personal-preference rules into a tool meant to work across arbitrary repos.

**Proposed fix.** Mark lessons applied from a single run record as **provisional** in the CHANGELOG entry (e.g. `(provisional, 1 occurrence — run: ...)`). Add a check to `claude-md-tidy-reflect` Step 1 (Gather evidence): if a later run's evidence *contradicts* a provisional lesson, that is itself top-priority evidence to revisit/demote it, cited the same way a fresh lesson would be. Promote a lesson to non-provisional once a second, independent run corroborates it.

**Competitive cross-check.** None of the nine surveyed repos have any equivalent self-correction mechanism — every competitor's rule set is either fully static (regex/heuristic lists in `geuneda`, `wrsmith108`, `Pinkers01`) or grows monotonically (`claude-reflect`'s correction capture). This item remains a genuine differentiator worth keeping, not something to downgrade based on competitive evidence.

**Files:** `skills/claude-md-tidy-reflect/SKILL.md` (Step 1, Step 5), `CHANGELOG.md` (entry format).
**Bump:** MINOR.

## 4. Duplicated bookkeeping surface (version + feature docs in multiple places)

**Problem.** The suite version lives in three places (two SKILL.md frontmatters + the README header); every feature is described in two places (SKILL.md normatively, README descriptively). This already produced one real sync slip during construction (a signal-numbering mismatch caught only by re-reading the file).

**Why it matters.** This is the exact "one home per rule" violation the hygiene rules warn against, self-inflicted. Accepted at the time as a deliberate tradeoff for a fully-documented README, but the drift risk is real and will compound with every reflection cycle.

**Proposed fix (narrow, not a redesign).** Keep the full README as-is (explicitly wanted), but tighten its role: README describes *behavior and intent* only, never restates a normative test or step verbatim — always a paraphrase or "see SKILL.md Step N" pointer. Add a line to `claude-md-tidy-reflect` Step 5 (Apply): when a lesson changes a SKILL.md step, the corresponding README bullet must be checked in the same diff, not just "if any documented feature changed" (which is easy to judge false when the change is subtle).

**Competitive cross-check.** None directly — this is an internal-documentation-hygiene finding with no counterpart in the competitor set (none of the nine repos maintain a parallel normative-doc/descriptive-doc split like this suite does).

**Files:** `README.md` (light rewording pass to paraphrase-not-duplicate), `skills/claude-md-tidy-reflect/SKILL.md` (Step 5 wording).
**Bump:** PATCH (tightens an existing step, no new capability).

## 5. CHALLENGE floods on a genuinely stale CLAUDE.md

**Problem.** Step 4 (Report and confirm) puts all CHALLENGE items first with no grouping or prioritization. A CLAUDE.md that has drifted far from reality could surface a long flat list of questions in one dump.

**Why it matters.** A wall of undifferentiated questions is exactly the attention-cost problem this suite exists to prevent, just relocated to the confirmation step instead of the file itself.

**Proposed fix.** Within CHALLENGE items, sort/group by stakes: rules that gate destructive or safety-relevant behavior (git operations, deploy, secrets) first, then rules three or more other lines depend on/reference, then everything else batched as "minor CHALLENGEs" the user can wave through as a block rather than one by one.

**Competitive cross-check.** `Pinkers01/claude-md-optimizer` ships a small hardcoded library of common rule-**conflict** shapes to check for explicitly (e.g. "autonomy vs. ask-first," "lite vs. full mode") rather than relying entirely on ad hoc discovery. The specific pairs it hardcodes are too repo-specific to adopt verbatim (they're tuned to its author's own projects — payment-provider choice, em-dash policy), but the *general mechanism* — a short, generalizable list of common contradiction *shapes* to actively probe for during Step 2b's line interrogation (the "Consistent?" test), rather than only surfacing a conflict when that check happens to trip over one — is worth adding alongside the grouping fix above, so CHALLENGE items arrive pre-sorted **and** some are caught proactively instead of only reactively.

**Files:** `skills/claude-md-tidy/SKILL.md` (Step 2b, Step 4).
**Bump:** MINOR.

## 6. No channel for feedback that arrives outside a tidy run

**Problem.** `RUNS.md` only gets a record when `/claude-md-tidy` itself runs, and `claude-md-tidy-reflect` Step 1 only picks up live-session feedback "if invoked right after a tidy run in this session." Feedback about CLAUDE.md quality or suite design that arises in an ordinary conversation *not* immediately following a tidy run (such as the review this plan is drawn from) never reaches `RUNS.md` and has no existing channel into `/claude-md-tidy-reflect`'s evidence-gathering.

**Why it matters.** Some of the best signal (including everything in this very document) comes from exactly this kind of ad-hoc, non-run conversation. Without a channel, that signal is lost unless someone manually re-derives it later.

**Proposed fix.** Extend `claude-md-tidy-reflect` Step 1 (Gather evidence) with a third source: an ad-hoc lesson described directly in the invoking conversation, with the conversation itself (and, where one exists, a doc like this plan file) standing in as the citation in place of a `RUNS.md` record.

**Competitive cross-check.** `BayramAnnakov/claude-reflect` splits this exact problem into an *automatic capture* hook (queues corrections the instant they happen, during any ordinary session) plus a *manual review* command (`/reflect`) that promotes queued items into permanent memory. That capture/review split is a concrete, already-proven implementation shape for this item: a lightweight hook that appends ad-hoc feedback to a queue automatically, reviewed and folded into `RUNS.md`-equivalent evidence the next time `claude-md-tidy-reflect` runs — rather than relying on someone remembering to describe the lesson in a future conversation.

**Files:** `skills/claude-md-tidy-reflect/SKILL.md` (Step 1).
**Bump:** MINOR.

## 7. No lightweight, LLM-free report/score entry point

**Problem.** The suite's only entry point is the full `/claude-md-tidy` workflow — repo survey, comprehension gate, per-line four-test interrogation, plan, confirm. There is no free, instant, no-plan-required way to just ask "how bad is this CLAUDE.md right now?"

**Why it matters.** Every direct competitor offers a cheap, no-commitment check: `geuneda/claude-md-optimizer` ships a standalone Python script (`analyze_claude_md.py`, with a `--json` mode) that scores 0-100 in a single non-agentic run; `tsalkin/claude-memory-hygiene`'s `report`/`lint` commands are explicitly read-only and free of any apply step. A user who just wants a quick signal before deciding whether a full tidy is worth running currently has no lighter-weight option in this suite.

**Proposed fix.** Add a `--report` (or equivalent) mode to `claude-md-tidy` that runs a cheap, mechanical pass only (no repo-comprehension survey, no per-line four-test verification) and stops there with no plan, no confirmation step, no edits: line/token count per file vs. the global hygiene guardrail; an approximate KEEP/COMPRESS/RELOCATE/DELETE/CHALLENGE tally from a shallow read; non-English/CJK content flagged with an estimated token-overhead cost (`geuneda/claude-md-optimizer`'s finding: non-English content runs 30-50% more tokens per instruction); and a session-cost estimate (per-request token cost × an assumed turn count, `geuneda`'s "session cost estimation"). All four are cheap, mechanical, non-judgment checks that fit a report mode's spirit — none require the four-test interrogation a full tidy does. Explicitly document the mode as a heuristic pre-check, not a substitute for a full tidy.

**Files:** `skills/claude-md-tidy/SKILL.md` (new mode/step), `README.md` (new row in the features table).
**Bump:** MINOR (new capability, additive, no change to the existing workflow contract).

## 8. No pre-relocate encryption-awareness check

**Problem.** The apply phase relocates content to new files (docs/, sibling skills) without checking whether the source CLAUDE.md — or the repo generally — is under git-crypt, SOPS, or age encryption.

**Why it matters.** `wrsmith108/claude-md-optimizer` calls out a specific failure mode this suite could hit silently: relocating content (including any encryption-unlock instructions) into an unencrypted sub-doc, or into an encrypted one that isn't readable without the very instructions being moved — a "chicken-and-egg" bug. This suite's own PRIMARY CHECK is about public-repo sensitive-content exposure; an encrypted repo is the adjacent case (content that's supposed to require a key to read, being moved somewhere that bypasses or breaks that requirement).

**Proposed fix.** Add an encryption-detection check to `claude-md-tidy` Step 0 (Preflight): scan for `.gitattributes` git-crypt/SOPS filters or an `.age` key reference. If detected, (a) force-classify any encryption-unlock instructions in the CLAUDE.md as un-relocatable (KEEP only), and (b) require any RELOCATE destination to be verified as covered by the same encryption scope as the source, not assumed.

**Files:** `skills/claude-md-tidy/SKILL.md` (Step 0 preflight, Step 5 apply-phase RELOCATE rule).
**Bump:** MINOR (new safety check, no workflow-contract change).

## 9. No pre-relocate CI/script-dependency scan

**Problem.** The apply phase greps for prose references to a moved term ("updates all referencing docs in the same change") but does not check whether any CI script or tooling regex-scans the CLAUDE.md file itself for specific content before relocating that content elsewhere.

**Why it matters.** `wrsmith108/claude-md-optimizer` treats this as a distinct constraint-detection phase specifically because a prose-reference grep and a script-dependency scan catch different things: a CI step that does `grep "required-phrase" CLAUDE.md` has no visible prose reference to the moved text, so the existing apply-phase grep would not catch it, and relocating that content would silently break the pipeline the next time it runs.

**Proposed fix.** Add a check to `claude-md-tidy` Step 0 (Preflight) or Step 5 (Apply): scan CI config (`.github/workflows/`, other CI dirs the repo survey already covers) for scripts that reference `CLAUDE.md` by filename, and treat any content those scripts appear to depend on as ineligible for RELOCATE without explicit user confirmation that the CI dependency is accounted for.

**Files:** `skills/claude-md-tidy/SKILL.md` (Step 0 or Step 5).
**Bump:** MINOR.

## 10. RELOCATE pointers have no "rich abstract" guidance

**Problem.** The suite's RELOCATE verdict leaves "a one-line pointer" at the original location with no guidance on what that pointer should contain beyond brevity.

**Why it matters.** `wrsmith108/claude-md-optimizer` ran an informal usage eval (5 sub-docs, 4 reference strategies, focused vs. ambiguous tasks) comparing reference styles: a thin link ("For X, see Y") gave an agent no way to judge relevance, so it opened all 5 of 5 sub-docs on the ambiguous task and 2 of 5 on the focused one; a **rich abstract** — a 3-5 sentence synopsis with concrete facts (framework, threshold, convention) plus the link — cut that to the main file only on the focused task (zero of the 5 sub-docs opened) and 2 of 5 on the ambiguous task. A pointer that's too thin to be self-sufficient defeats the purpose of relocating: the agent ends up reading the sub-doc anyway on every task that isn't perfectly unambiguous, so no context is actually saved during the session (only at startup).

**Proposed fix.** Revise the RELOCATE verdict's pointer requirement from "a one-line pointer" to "a synopsis that states the concrete fact(s) a reader would otherwise need to open the sub-doc for (the specific convention, threshold, or command — not just that a sub-doc exists), plus the link" — still short, but self-sufficient for the common case rather than a bare cross-reference.

**Files:** `skills/claude-md-tidy/SKILL.md` (RELOCATE verdict definition, Step 5 apply-phase relocate step).
**Bump:** PATCH (tightens existing verdict wording, not a new verdict or workflow step).

## 11. The suite's own `RUNS.md` has no archive/pruning mechanism

**Problem.** `RUNS.md` (run records for `claude-md-tidy-reflect`) is, per this repo's own README, "created on first run, newest at top" — and accumulates a new entry every time `/claude-md-tidy` runs, forever. `claude-md-tidy-reflect` marks consumed records `Processed: yes (vX.Y.Z)` but nothing ever removes or archives a processed record.

**Why it matters.** This is the same "the bloat-fighting tool accumulates its own bloat" irony as item 1, but at the *data* layer instead of the *rules* layer — and it is precisely the failure mode `tsalkin/claude-memory-hygiene` was built to fix for Claude Code's `MEMORY.md`: an always-referenced index file that only ever grows, eventually either exceeding a practical size or making evidence-gathering (`claude-md-tidy-reflect` Step 1, which must read `RUNS.md`) increasingly expensive for no benefit once a record's lesson has been fully processed and applied.

**Proposed fix.** Borrow `tsalkin`'s pin/volatile/archive model, adapted to this suite's own vocabulary: once a `RUNS.md` record is marked `Processed: yes` **and** was not tagged `provisional` (per item 3) in a way that's still pending corroboration, it becomes an archive candidate. Add an archive step to `claude-md-tidy-reflect` Step 5 (Apply) that moves fully-processed, non-provisional records older than the N most recent into a dated file inside a `RUNS-archive/` directory (never deletes), leaving `RUNS.md` itself bounded to recent/active records plus anything still provisional or pinned.

**Files:** `skills/claude-md-tidy-reflect/SKILL.md` (Step 5), `README.md` (file-layout section, to document the `RUNS-archive/` directory).
**Bump:** MINOR (new capability, additive).

## 12. No deterministic backstop against runaway growth between tidy runs

**Problem.** TidyClaudeMD's slimming is entirely judgment-driven and only runs when explicitly invoked. Nothing detects or warns when a target repo's CLAUDE.md crosses the global hygiene rule's own ~150-line guardrail *between* tidy runs — the suite only ever finds out the next time someone happens to invoke `/claude-md-tidy`.

**Why it matters.** `alirezarezvani/ClaudeForge` enforces a hard 150-line cap **deterministically**, via a `PostToolUse(Edit|Write)` hook, independent of any LLM judgment call — catching drift the moment it happens rather than waiting for a manual invocation. This suite's four-test, judgment-driven approach is deliberately more nuanced than a hard cap (a considered design choice, not an oversight — see the "Considered and rejected" section below), but a purely *advisory* version of the same mechanism (a hook that surfaces a reminder, not a hard block) would close the gap between "drift happened" and "someone noticed" without compromising the judgment-driven philosophy.

**Proposed fix.** Document (not necessarily ship, since hook installation is a repo-level, per-user decision) an optional `PostToolUse` hook pattern users can adopt: on Edit/Write to a CLAUDE.md, check line count against the global hygiene guardrail and print an advisory reminder (not a block) to run `/claude-md-tidy` if crossed. Keep this as guidance/an example, not a mandatory part of the skill workflow — a hard block would conflict with the suite's core "nothing is edited before the user confirms" invariant applied in reverse (a hook silently blocking an edit is its own kind of unconfirmed intervention).

**Files:** `README.md` (new "optional companion hook" section, with an example `hooks.json` snippet).
**Bump:** MINOR (new documented capability, opt-in, no change to the skills themselves).

## 13. No injection-order awareness in the KEEP/COMPRESS placement judgment

**Problem.** Step 3 (Analyze) classifies each block by verdict but has no test for *where* KEEP/COMPRESS content should sit given Claude Code's actual config load order (global CLAUDE.md → global rules → project CLAUDE.md → project rules → Memory). A project CLAUDE.md restating a global prohibition verbatim currently has no verdict pathway other than DELETE-as-duplicate (if found identically elsewhere) or CHALLENGE (if it's a near-duplicate) — there's no test for "this is redundant specifically because injection order already surfaces it, even if the wording differs."

**Why it matters.** `geuneda/claude-md-optimizer` treats injection-order-aware placement as a distinct analysis pass and flags exactly this case — a global prohibition repeated in a project file that's "already well-attended via injection order" — as bloat independent of exact-duplicate matching. Without this, TidyClaudeMD's DELETE verdict (which requires a cited exact-duplicate location or a grep-confirmed dead reference) can miss same-intent-different-wording redundancy that a load-order-aware check would catch.

**Proposed fix.** Add a fifth question to Step 2b's line interrogation, alongside True/Live/Consistent/Actionable: **Redundant-by-order?** — does a global-scope file (or an earlier-loaded rule) already establish this same intent, making a project-scope restatement pure repetition regardless of exact wording? A line failing this test routes to COMPRESS (trim to a one-line acknowledgment) or DELETE (if the restatement adds no project-specific nuance at all), decided the same evidence-first way the existing DELETE rule works: cite the earlier-loaded rule it's redundant with.

**Files:** `skills/claude-md-tidy/SKILL.md` (Step 2b new question, Step 3 verdict table note).
**Bump:** MINOR (new interrogation test, extends existing verdict logic — not a new verdict).

## 14. No detection of gitignored personal-override CLAUDE.md files

**Problem.** Step 2's target-finding bullet is `git ls-files '*CLAUDE.md' 'CLAUDE.md'` plus "an untracked-file check." Neither surfaces a **gitignored** file: `git ls-files` never lists ignored paths, and a standard untracked-file check (`git status --porcelain`) excludes them too by default. A personal-override convention like `alirezarezvani/ClaudeForge`'s `CLAUDE.local.md` — explicitly "auto-gitignored" — would be invisible to this step entirely.

**Why it matters.** Such a file is still loaded into every session exactly like any other CLAUDE.md, so it's squarely within this suite's stated promise to "sweep every CLAUDE.md in the repo, root and nested." A repo using this (or any similar gitignored-override) convention would have a whole category of always-loaded instructions this suite silently never finds, tidies, or even reports as skipped.

**Proposed fix.** Extend Step 2's target-finding to also glob the filesystem directly for common personal-override filenames (starting with `CLAUDE.local.md`) rather than relying on `git ls-files`/untracked-check alone, and list any found in the Step 4 plan explicitly — even if the user then chooses to exclude personal files from tidying, that should be a stated choice, not a silent gap.

**Files:** `skills/claude-md-tidy/SKILL.md` (Step 2, target-finding bullet).
**Bump:** MINOR (extends existing target-finding, no workflow-contract change).

## Considered and rejected

- **Cross-file duplicate detection across global/project/rules files** (`geuneda/claude-md-optimizer`). Already covered, not a gap: the existing DELETE verdict ("duplicated verbatim elsewhere, cite location") plus Step 2b's **Consistent?** test already catch exact and near-exact duplicates across files in scope. Checked explicitly against `plans/competitors/geuneda-claude-md-optimizer.md` and found no distinct capability to add beyond item 13's load-order-aware refinement above.
- **Hard numeric line-count verdicts** (`NicoAcosta/claude-md-optimizer`'s <50/100-line hard target; `geuneda/claude-md-optimizer`'s per-file-type numeric table). Rejected as an in-suite mechanism: a context-free numeric threshold is exactly the kind of rule the suite's own four-test interrogation (True/Live/Consistent/**Actionable**) exists to catch when it shows up *inside* a CLAUDE.md — adopting one as the suite's own verdict logic would be inconsistent with that standard. The global hygiene rule's "~150 lines" is deliberately a soft guardrail that *triggers* a tidy, not a hard pass/fail line the skill itself enforces — item 12 above proposes making that trigger deterministic and visible, without turning it into a verdict rule.
- **Hardcoded, repo-specific contradiction pairs** (`Pinkers01/claude-md-optimizer`'s Stripe-vs-Mollie, em-dash-policy style list). Rejected verbatim: tuned to one author's own recurring situations, not generalizable across arbitrary repos. The generalized version — a short list of common contradiction *shapes* to actively probe for — is folded into item 5 instead.
- **Multi-format support** (AGENTS.md, copilot-instructions.md — `wrsmith108`, `NicoAcosta`). Noted as a real scope difference (both competitors cover formats this suite doesn't) but explicitly out of scope for this backlog: broadening format coverage is a product/strategy decision about what this suite is *for*, not a mechanical fix to an existing gap, and deserves its own explicit user decision rather than being folded in here.
- **Building an in-suite structural/schema linter** (`agent-sh/agnix`'s 432 correctness rules across six+ agent-tool formats). Rejected: orthogonal problem (syntax/schema correctness vs. content bloat) already well covered by an existing, actively maintained, IDE-integrated tool. Better handled by a README pointer recommending agnix as a complementary pass than by rebuilding linting logic in-suite.

## Not fixable by design — noted, not actioned

**No way to measure whether a self-improvement actually improved anything.** The invariant gate limits *damage* from a bad lesson; nothing measures *benefit*. There is no eval harness and none is proposed here — this is an inherent limit of judgment-driven self-modification, mitigated (not solved) by this repo's git history giving a rollback path. (No competitor in the surveyed set has a working eval harness for this either — `wrsmith108`'s "measured results" section is the closest, and it explicitly notes its own harness "is not yet committed.")

## Suggested execution order

**Tier 1 — trust that the loop doesn't silently degrade or grow unbounded (do first, in any order):** items 1, 3, 11. These are all about whether repeated self-improvement cycles can be trusted to run indefinitely without either baking in bad lessons (1, 3) or accumulating unbounded bookkeeping (11).

**Tier 2 — bound cost before this suite meets a large/production target:** items 2, 7. Item 2 bounds the cost of a *full* tidy; item 7 gives a cheap, deliberately separate alternative to running one at all (item 7's mechanical pass explicitly skips the survey machinery item 2 improves, so the two are independent fixes worth doing in the same window, not a shared implementation).

**Tier 3 — safety checks before the apply phase touches content:** items 8, 9. Both are preflight/apply-phase checks that prevent a relocate from silently breaking something (an encrypted file, a CI dependency) outside the CLAUDE.md itself.

**Tier 4 — low-risk, independently shippable at any time:** items 4, 5, 6, 10, 12, 13, 14. None of these touch the six protected invariants and none depend on any other item landing first.

None of these fourteen items require the invariant-gate escalation defined in the README — ordinary user confirmation before applying is sufficient for all of them.
