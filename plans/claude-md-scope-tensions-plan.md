## The five axes

Every line in a CLAUDE.md implicitly claims to belong exactly where it sits — and, per axes 4 and 5, entire *other files* can carry rules that a tidy of the literal CLAUDE.md never sees. Five related questions (they overlap at the edges — axis 4's decision rule is partly axis 1 applied to imports — but each has its own mechanism and its own failure mode) can each falsify that claim:

1. **CLAUDE.md vs. Skill** — is this a fact needed every session, or a procedure/reference used only sometimes?
2. **Project CLAUDE.md vs. user/global CLAUDE.md vs. `CLAUDE.local.md`** — is this project-specific, cross-project-personal, or repo-local-personal?
3. **CLAUDE.md vs. Memory (`MEMORY.md`)** — is this a deliberate, human-authored rule, or an agent-discovered pattern?
4. **CLAUDE.md's `@import` mechanism** — does this CLAUDE.md pull in other files (of which `@AGENTS.md` is one common instance, not a special case), and if so, is the skill actually seeing what those imports contain?
5. **CLAUDE.md vs. `.claude/rules/`** — should this content be a modular rule file, in particular a *path-scoped* rule that only loads when Claude works with matching files?

`claudemd-tidy` today only reasons well about axis 1 (via RELOCATE), plus the one sub-case of axis 2 that Redundant-by-order already catches (a project line duplicating an earlier-loaded global rule). Everything else on axes 2–5 is unhandled — a line can be flagged True/Live/Consistent/Actionable/non-redundant and still be filed in the wrong *scope* entirely (or, per axes 4–5, an entire sibling file's content can be invisible to the audit), and nothing currently catches that.

## Axis 1: CLAUDE.md vs. Skill — research

Official guidance (`features-overview.md`, `best-practices.md`, `memory.md`):

- **Size ceiling**: "target under 200 lines per CLAUDE.md file" — bloated files "consume more context and reduce adherence." (This project's own global hygiene rule already uses a stricter internal ~150-line guardrail — a deliberate, tighter-than-official convention, not a discrepancy to fix.) Note the tooling side moved past a fixed number: since Claude Code 2.1.169, "the 'CLAUDE.md is too long' warning threshold now scales with the model's context window" — the doc's ~200-line figure is writing guidance, not the warning trigger.
- **Context-cost asymmetry**: CLAUDE.md's full content loads every session; a Skill's description loads every session but its full content only loads when invoked or matched to the task ("progressive disclosure"). This is the mechanical reason procedural/reference content is expensive in CLAUDE.md and cheap in a Skill.
- **Decision rule** (official, current `memory.md` wording): a fact Claude should know *every* session (build commands, conventions, "never do X" rules) → CLAUDE.md. "If an entry is a multi-step procedure or only matters for one part of the codebase, move it to a skill **or a path-scoped rule** instead" (emphasis added). The official rule now names two destinations, not one — the path-scoped-rule half is axis 5's subject.

**Gap check against the current skill:** `claudemd-tidy`'s RELOCATE verdict test is *"Procedure/step detail whose natural owner is a skill, `docs/`, or README"* — this matches the skill half of the official decision rule closely. **The skill half of axis 1 is already substantially covered**; the path-scoped-rule half is not (no rules destination exists — see axis 5, item 5). One small, low-priority refinement remains here: the RELOCATE test could cite the official framing directly ("only matters for one part of the codebase," "used sometimes, not every session") to sharpen borderline calls — a wording tightening folded into item 1 below, not a new mechanism.

## Axis 2: Project vs. user/global vs. local CLAUDE.md — research

Official guidance (`memory.md`, "Choose where to put CLAUDE.md files" scope table and "How CLAUDE.md files load"):

| Scope | Location | Purpose | Shared with |
|---|---|---|---|
| Managed policy | `/Library/Application Support/ClaudeCode/CLAUDE.md` (macOS), `/etc/claude-code/CLAUDE.md` (Linux/WSL), `C:\Program Files\ClaudeCode\CLAUDE.md` (Windows), or a `claudeMd` key in `managed-settings.json` | Org-wide instructions managed by IT — "loads before user and project CLAUDE.md," "cannot be excluded" | All users in the organization |
| User instructions | `~/.claude/CLAUDE.md` | Personal preferences for **all** projects | Just you |
| Project instructions | `./CLAUDE.md` or `./.claude/CLAUDE.md` | Team-shared instructions | Team, via source control |
| Local instructions | `./CLAUDE.local.md` (gitignored) | Personal, **project-specific** preferences | Just you, this project only |

The managed-policy row is IT-deployed and impossible to exclude — it is never a tidy *target* or *destination*; it matters here only as an earlier-loaded scope that Redundant-by-order should account for when such a file exists on the machine.

Key mechanics: all scopes are **additive, not overriding** — every discovered file concatenates into context (root-down by directory, `CLAUDE.local.md` appended after `CLAUDE.md` at each level). Nested per-directory CLAUDE.md files are a documented, first-class pattern for monorepos, loaded on demand when Claude reads files in that subdirectory.

**Decision rule** (official): personal, cross-project habits → user `~/.claude/CLAUDE.md`. Team-shared, project-specific conventions → project `CLAUDE.md`, git-tracked and PR-reviewed. Personal preferences specific to *this* project only (sandbox URLs, personal branch names, local test data) → `CLAUDE.local.md`, gitignored.

**Gap check against the current skill:** one sub-case is covered, the rest is not. A project line that *duplicates* an earlier-loaded global rule is already caught by Redundant-by-order. But a project `CLAUDE.md` line that is a generic, cross-project working preference **not yet in the global file** has no test that would flag it — nothing says "true, live, consistent, actionable, non-redundant, and still in the wrong scope." Same for a personal, repo-specific preference someone hardcoded into the shared, team-reviewed `CLAUDE.md` instead of `CLAUDE.local.md`. **This is the concrete gap the user identified in conversation** — proposed fix below (item 2).

## Axis 3: CLAUDE.md vs. Memory (`MEMORY.md`) — research

Official guidance (`memory.md`, "CLAUDE.md vs auto memory"):

| Aspect | CLAUDE.md | Auto Memory |
|---|---|---|
| Who writes it | You (human-authored) | Claude (agent-discovered) |
| Contains | Instructions and rules | Learnings and patterns |
| Loaded into context | Full content, every session | Index only (`MEMORY.md`, capped at 200 lines / 25KB) — topic files recalled on demand, not preloaded |
| Use for | Coding standards, workflows, architecture | Build commands Claude discovered, debugging insights, preferences Claude noticed |

Neither CLAUDE.md nor Memory *enforces* anything — both are context that shapes reasoning, not a hard gate (for hard guarantees, official guidance points to `PreToolUse` hooks instead). The documented **promotion path**: when something Claude learned in Memory turns out to matter every session, the user says "add this to CLAUDE.md" (or edits directly) — moving it from agent-discovered-and-ephemeral to human-decided-and-durable.

**Decision rule** (official): deliberate, stable, team-relevant rules → CLAUDE.md. Ad hoc discovered patterns, debugging insights, machine-local tool preferences → Memory, not CLAUDE.md.

**Gap check against the current skill:** entirely unhandled. A CLAUDE.md line that reads like a discovered debugging insight or an ad hoc noticed pattern (not a deliberate team rule) rather than an intentional instruction is invisible to all five current verdicts — True/Live/Consistent/Actionable/Redundant-by-order all pass a line like this if it happens to be accurate and unambiguous, even though it's arguably the wrong *kind* of content for a PR-reviewed rulebook. Proposed fix below (item 3) — deliberately scoped narrower than the full bidirectional promotion path, for reasons explained there.

## Axis 4: CLAUDE.md's `@import` mechanism (general), with AGENTS.md as one instance — research

Official guidance, verified directly against the source text of `memory.md` "Import additional files" and "AGENTS.md" sections (not just the research agent's paraphrase — see the third-pass update note above for why):

- **`@path` import is fully general-purpose, not AGENTS.md-specific.** Verbatim: *"CLAUDE.md files can import additional files using `@path/to/import` syntax."* Documented examples include `@README`, `@package.json`, `@docs/git-instructions.md`, and `@~/.claude/my-project-instructions.md` — arbitrary files, anywhere in the CLAUDE.md body, not just at the top.
- **AGENTS.md is explicitly the special case, not the general mechanism**: *"Claude Code reads `CLAUDE.md`, not `AGENTS.md`."* The documented bridge is exactly this same `@import` syntax (`@AGENTS.md`, with Claude-specific content appended below) or a symlink (`ln -s AGENTS.md CLAUDE.md`, Unix-only). No Claude Code changelog entry has ever mentioned AGENTS.md (checked through v2.1.204) — the import/symlink bridge is the whole story.
- **Recursive, depth-limited**: *"Imported files can recursively import other files, with a maximum depth of four hops."* **Correction from the first research pass**: no cycle detection is documented for `@import` specifically — the doc's only cycle-detection statement (*"circular symlinks are detected and handled gracefully"*) is about `.claude/rules/` symlinks, a different mechanism (see axis 5). The four-hop depth cap is the actual, confirmed safety bound for imports; don't assume cycle detection beyond that.
- **Path resolution**: both relative and absolute paths allowed; relative paths resolve **relative to the file containing the import**, not the working directory or repo root. Home-directory (`~`) paths are supported and are the documented way to share personal instructions across git worktrees (a gitignored `CLAUDE.local.md` doesn't follow you between worktrees; a `~`-path import does).
- **No context-cost saving**: *"Splitting into `@path` imports helps organization but does not reduce context, since imported files load at launch."* An imported file's content is exactly as context-expensive as the same lines written inline — it just lives elsewhere. (Corollary, deliberately left open below: whether the ~150-line hygiene guardrail should be measured against the *effective*, import-inclusive size.)
- **Approval gate — for *external* imports specifically.** Verbatim: *"The first time Claude Code encounters **external imports** in a project, it shows an approval dialog listing the files. If you decline, the imports stay disabled and the dialog does not appear again."* (emphasis added) Changelog 2.1.6 uses the same scoping ("the external CLAUDE.md imports approval dialog"). So an import pointing **outside the project** (a `~`-path, an absolute path elsewhere) can be syntactically present yet **permanently inert** for a given user/machine if it was declined once — while an in-project import (`@AGENTS.md`, `@docs/git-instructions.md`) is not subject to this gate and can be assumed live if syntactically valid.
- **Literal mentions are excluded, not just any `@word`.** Verbatim: *"Import parsing skips Markdown code spans and fenced code blocks. To mention a path in your CLAUDE.md without importing it, wrap it in backticks."* `` `@README` `` in backticks is inert text; `@README` outside backticks is a live import. Any detection logic has to make this distinction or it will misclassify prose that merely *mentions* an import syntax as if it were one. (Caution: the changelog also records parser edge cases beyond the documented rules — e.g. 2.0.67 fixed consecutive `@~/` references being misparsed as markdown strikethrough — so treat any detection as best-effort pattern-matching against the documented rules, not a reimplementation of the real parser.)
- **Inspection**: `/memory` "lists all CLAUDE.md, CLAUDE.local.md, and rules files loaded in your current session" — the current doc wording doesn't name arbitrary `@`-imported non-CLAUDE.md files (a `README`, a `package.json`), though changelog 1.0.94 ("/memory command now allows direct editing of all imported memory files") suggests imported files were surfaced there at least historically. Treat full import visibility in `/memory` as likely-but-unconfirmed. Separately, the `InstructionsLoaded` hook (added 2.1.69; now recommended by the docs for exactly this) fires when instruction files load into context — a first-party way to *log* what actually loaded, though only if such a hook is installed before the session.

**Skills, by contrast, have no equivalent import mechanism at all — confirmed by direct read of `skills.md` "Add supporting files."** Supporting files (`reference.md`, `examples.md`, `scripts/helper.py` — no mandated folder name, any layout works) are referenced from `SKILL.md` via **plain markdown links** (`[reference.md](reference.md)`), read by Claude only when it decides they're relevant — strictly lazy, never eagerly loaded the way a CLAUDE.md `@import` is. `SKILL.md` itself has its own, looser size guidance: *"Keep SKILL.md under 500 lines."* This means axis 4's specific tension (an eagerly-loaded, always-active import) **does not exist for Skills** — there's nothing analogous for `claudemd-tidy` to audit on the Skills side beyond what axis 1 already covers.

**Decision rule** (official): a repo supporting multiple AI coding tools keeps shared instructions in `AGENTS.md` (or any other shared file) and imports it into `CLAUDE.md`, rather than maintaining duplicated copies. More generally: anything genuinely needed every session belongs in a `CLAUDE.md` import; anything used only sometimes belongs in a Skill's lazily-loaded supporting file instead (this is axis 1's rule applied to the choice of *how* to pull in extra content, not just whether to).

**Gap check against the current skill:** three distinct gaps, all unhandled today, sharing one root cause — the skill doesn't currently look for or follow any `@import`, AGENTS.md or otherwise:

(a) **Repo-level structural gap.** Step 2's target-finding (bullet 1) looks for `*CLAUDE.md`/`CLAUDE.md` and `CLAUDE.local.md`-style files; it never checks whether any found file contains `@import` lines, nor what they point to. A repo with an `AGENTS.md` (or any other instruction file) that no `CLAUDE.md` imports has a real, concrete consequence beyond this skill: Claude Code itself never reads that content in *any* session. Two sub-cases: no `CLAUDE.md` exists at all (Claude Code has zero project-specific context even if other tools do, via `AGENTS.md` or otherwise), or a `CLAUDE.md` exists but doesn't import the sibling file (blindness, plus drift risk if both independently maintain overlapping rules).

(b) **Import-following gap.** When a `CLAUDE.md` *does* contain a live `@import` (correctly distinguished from a backtick-escaped mention, per the finding above), the imported content is exactly as "loaded every session" as anything written directly in `CLAUDE.md` — but Step 2b's line-by-line interrogation only covers the target CLAUDE.md's own text today, so imported content is never audited despite being fully active in every session. A second, smaller instance of the same blindness: Step 1 reads the **global** `~/.claude/CLAUDE.md` for the Redundant-by-order comparison — if the global file itself contains live imports, the redundancy comparison silently misses whatever those imports contribute.

(c) **Inert-import gap — scoped to external imports.** An import pointing outside the project can be syntactically present yet inactive if the user declined the one-time approval dialog (in-project imports are not gated — see the research bullet above). The skill can't detect the declined state from file content alone (an `InstructionsLoaded` hook could log actual loads, but only as an optional, pre-installed companion — same advisory pattern as the README's existing `PostToolUse` hook, not something the skill can rely on). At minimum the skill must *know this possibility exists* for external imports rather than assume every written `@import` line is live — otherwise a CHALLENGE or audit finding could confidently reference imported content that isn't actually reaching the model in session.

## Axis 5: CLAUDE.md vs. `.claude/rules/` — research

Added in the fourth pass (user decision, 2026-07-08) after the changelog comparison showed this is a mature, actively developed scope the plan had missed: added 2.0.64, conditional `paths:` loading and print-mode support 2.1.69, YAML glob lists 2.1.84, symlinked-path matching fixed as recently as 2.1.198. Verified directly against `memory.md` "Organize rules with `.claude/rules/`":

- **Modular rule files**: *"For larger projects, you can organize instructions into multiple files using the `.claude/rules/` directory."* All `.md` files are discovered recursively (subdirectories like `frontend/` work); each file is meant to cover one topic with a descriptive filename.
- **Unconditional rules are CLAUDE.md-equivalent**: *"Rules without `paths` frontmatter are loaded at launch with the same priority as `.claude/CLAUDE.md`."* They are always-loaded context, exactly like CLAUDE.md body text.
- **Path-scoped rules are the genuinely new capability**: YAML frontmatter `paths:` (glob patterns, brace expansion, multiple patterns) makes a rule load *"only when Claude works with matching files, reducing noise and saving context space."* Triggering is on file-read, not every tool use; since 2.1.198 it also matches through symlinked paths.
- **Placement guidance (official)**: *"Rules load into context every session or when matching files are opened. For task-specific instructions that don't need to be in context all the time, use skills instead."* And from the CLAUDE.md size guidance: *"If your instructions are growing large, use path-scoped rules so instructions load only when Claude works with matching files."*
- **User-level rules**: `~/.claude/rules/` applies to every project on the machine; *"user-level rules are loaded before project rules, giving project rules higher priority."* Another earlier-loaded scope Redundant-by-order should account for.
- **Symlink sharing**: rules directories/files can be symlinked from shared locations; *"circular symlinks are detected and handled gracefully"* — this is the sentence the first research pass mis-attributed to `@import`.

**Decision rule** (official, three-way): needed every session → CLAUDE.md (or an unconditional rule file — same load behavior, modular file). Needed only when working with certain paths → **path-scoped rule**. Needed only for specific tasks, on invocation or task-match → **skill**. This completes axis 1's rule: "only matters for one part of the codebase" officially routes to a path-scoped rule *or* a skill, and the path-scoped rule is the better fit when the content is a fact/constraint about those files rather than a procedure.

**Gap check against the current skill:** unhandled on both the read and the write side. Read side: Step 2's survey never looks at `.claude/rules/` — but unconditional project rules are always-loaded context with CLAUDE.md-equal priority, so a contradiction or duplication between a CLAUDE.md line and a rules file is exactly the kind of thing Consistent? and Redundant-by-order exist to catch, and today they can't see it. Write side: RELOCATE's destination inventory (Step 2 bullet 4: commands, skills, `docs/`, `plans/`, README) doesn't include `.claude/rules/` at all, so path-scoped content currently RELOCATEs to a skill even where a path-scoped rule is the documented better home. Proposed fix below (item 5).

## Proposed changes to `claudemd-tidy`

### 1. Axis 1 — sharpen RELOCATE's citation (low priority, optional)

**Proposed fix.** Add the official framing directly to the RELOCATE test in the Step 3 verdict table: *"...or content that only matters for one part of the codebase, or is used occasionally rather than every session (official Claude Code guidance: CLAUDE.md is for facts needed every session; Skills and path-scoped rules are for content used only sometimes or only for some files)."* This doesn't change behavior — RELOCATE already catches these cases — it just gives the model a citable, sharper rule of thumb for borderline calls between "leave as COMPRESS" and "RELOCATE."

**Files:** `skills/claudemd-tidy/SKILL.md` (Step 3 verdict table, RELOCATE row). Note: item 5 edits the same row — if both ship, land them together as one edit.
**Bump:** PATCH (wording sharpened, no new capability).

### 2. Axis 2 — new Step 2b test: **Correctly-scoped?**

**Proposed fix.** Add a sixth Step 2b question: *"Correctly-scoped? Is this line's content actually about **this repo** — or is it a generic, cross-project working preference (→ belongs in user `~/.claude/CLAUDE.md`) or a personal-but-repo-specific preference not relevant to the rest of the team (→ belongs in `CLAUDE.local.md`, gitignored)?"*

**Precedence with Redundant-by-order:** Correctly-scoped? fires only when the content is **not** already established in an earlier-loaded scope — if the global file already covers it, that's the existing Redundant-by-order path (COMPRESS or CHALLENGE per the current table), not a scope question. The two tests are mutually exclusive by construction: Redundant-by-order handles "this already exists elsewhere," Correctly-scoped handles "this exists only here but belongs elsewhere."

Routing: failing this test alone (the line is otherwise True/Live/Consistent/Actionable/non-redundant) is not "wrong," just misplaced — route to **CHALLENGE**, never straight to RELOCATE or DELETE, since only the user can confirm whether something is genuinely meant to be universal/personal versus deliberately pinned to this one project. Extend the CHALLENGE option set presented in Step 4 (`fix to match reality / keep — I'm missing context / delete`) with two more: *promote to global `~/.claude/CLAUDE.md`* / *move to `CLAUDE.local.md`* (creating `CLAUDE.local.md` in the apply phase if it doesn't exist, and confirming it's gitignored).

**Apply-phase discipline for out-of-repo writes (decision resolved with the user, 2026-07-08 — no new invariant).** "Promote to global `~/.claude/CLAUDE.md`" writes *outside* the repo, into a file shared across every project the user has. Resolution: this is governed by **documented apply-phase discipline, not a seventh protected invariant** — existing invariant 1 ("no CLAUDE.md is edited before the user approves the plan") already covers the global file by its own wording, so the gap is only *bundling*, not approval. The rule, stated here once and referenced by item 4: **any resolution that writes outside the repo being tidied is confirmed individually at the Step 4 stop point — named file, exact content — never bundled into a bulk "approve all"** (an in-repo RELOCATE may be bulk-approved; an out-of-repo write may not). This matches the discipline `claudemd-tidy-reflect` already applies to global-rulebook edits (its Step 3: propose the edit, get explicit confirmation). Accepted residual risk of the no-invariant choice: a future reflect pass could weaken this rule without tripping the invariant gate — mitigated by it being an explicit, documented rule a pruning lesson would have to cite evidence against.

**Files:** `skills/claudemd-tidy/SKILL.md` (Step 2b new question, Step 4 CHALLENGE option list + individual-confirmation rule, Step 5 apply phase — new destination cases: global CLAUDE.md, `CLAUDE.local.md` creation).
**Bump:** MINOR per the suite's table ("new step, verdict, signal, or capability" — a new Step 2b test is a new capability; MAJOR is reserved for workflow-contract changes, which this isn't: same phases, same stop point).

### 3. Axis 3 — new Step 2b test: **Rule-vs-memory?** (one direction only, deliberately)

**Proposed fix.** Add a seventh Step 2b question: *"Rule-vs-memory? Does this line read as a deliberate, team-decided instruction — or as a discovered pattern, debugging insight, or ad hoc noticed preference that reads more like something Claude learned than something the team decided?"*

Routing: failing this test alone routes to **CHALLENGE**, with the question surfaced as *"this reads like a discovered pattern rather than a deliberate rule — keep it here, or does it belong in your memory system instead?"* The two resolutions and exactly what each does:

- **Keep** — the user says it's deliberate; the line stays, final verdict KEEP (record the confirmation in the run record).
- **Belongs in memory** — the skill **does not** write to memory and **does not** remove the line in this run. It leaves the line untouched, and notes in the Step 6 report that the user intends to move it (the move itself is the documented manual path: ask Claude in a normal session, or edit memory directly). Only once the content verifiably exists in memory — e.g. a later tidy run greps `MEMORY.md`/topic files *for that specific content* (a targeted claim-verification, consistent with Step 2's claim-driven survey style — not the open-ended memory reading the second reason below declines) — does the original line become a cited duplicate eligible for the normal DELETE evidence rule. This keeps the no-loss invariant intact: the skill never removes a line whose destination it didn't write and can't verify.

**Deliberately no auto-apply and no writes into `~/.claude/projects/<slug>/memory/`.** Two reasons:

- Memory files routinely accumulate personal data (the user's own global PRIMARY CHECK rule treats "anything copied from private memory" as sensitive by default) — writing into that location adds a privacy surface the skill doesn't currently touch anywhere else.
- The reverse direction — reading Memory to find content that should be *promoted* to CLAUDE.md — would require `claudemd-tidy` to read a completely new artifact class its survey has no relationship with today, and that promotion path is already officially documented as a manual, user-initiated action ("ask Claude to add this to CLAUDE.md, or edit the file yourself"). Not worth building a redundant path for something the platform already handles.

This keeps the check useful (flags miscategorized content) without expanding the skill's read/write surface into a new, more sensitive artifact type.

**Superseded-in-part by the mission plan (2026-07-08).** `plans/instruction-surface-maintenance-plan.md` makes auto memory a full read+write maintenance target, with reversibility via per-run snapshots rather than git. Once that capability ships (its Phase 2), the "belongs in memory" resolution here upgrades from manual handoff to a skill-performed, snapshot-protected move — the CHALLENGE routing and the user-decides principle are unchanged; only the execution mechanics improve. Until then, the manual-handoff semantics above are the spec.

**Files:** `skills/claudemd-tidy/SKILL.md` (Step 2b new question, Step 4 CHALLENGE option list, Step 6 report — the "user intends to move" note).
**Bump:** MINOR per the suite's table (new test = new capability; not a workflow-contract change).

### 4. Axis 4 — detect and follow `@import` lines generally (AGENTS.md is one case, not the mechanism)

**Proposed fix (three parts, one item — all stem from the same root cause: the skill doesn't currently look for or follow any `@import`).**

**Part a — detect unimported sibling files.** Extend Step 2 bullet 1 to also check for `AGENTS.md` specifically (the one documented, named cross-tool convention worth checking for by default) at the repo root and nested. If found, check whether any `CLAUDE.md` in the repo pulls it in — via a live `@import` (not a backtick-escaped mention; this needs part c's requirement 1, so part a and that requirement ship together) or a symlink (`CLAUDE.md → AGENTS.md`). The only direction that matters is CLAUDE.md importing AGENTS.md: Claude Code doesn't read AGENTS.md, and other tools don't parse `@import` syntax, so "AGENTS.md references CLAUDE.md" changes nothing. Because an in-repo import is not approval-gated (see research), a detected live in-repo import can be asserted as active without hedging. Surface the result in the Step 4 plan as an explicit repo-level note, the same treatment already given to the `CLAUDE.local.md` finding: *"Found AGENTS.md at `<path>` — [not imported by any CLAUDE.md / imported by `<file>`]. Claude Code only reads CLAUDE.md directly; an unimported AGENTS.md is invisible to every Claude Code session, not just this tidy run."* Never auto-adds an import line — always surfaces as information at minimum, escalates to CHALLENGE when it looks like unintentional drift (both files exist, no CLAUDE.md imports the AGENTS.md, and overlapping rule text is found in both). Note this is a **repo-level** CHALLENGE — a deliberate extension of a verdict that is per-block today; the SKILL.md edit should say so explicitly rather than leaving CHALLENGE's semantics silently widened. To also run this during `--report` mode (worthwhile — it's cheap and report mode is exactly the "how bad is it?" pre-check where an invisible AGENTS.md matters), the report-mode section needs its own hook for it, since `--report` skips Step 2 entirely.

**Part b — follow *any* live import during interrogation, not just AGENTS.md.** For every live `@path` import found in a target `CLAUDE.md`, treat the imported file's content as part of the "effective" CLAUDE.md for the *audit*: Step 2b's line-by-line walk includes the imported file's lines, and Step 3's verdicts can apply to blocks that physically live there. Respect the documented four-hop recursion limit — don't follow a chain deeper than that, and don't assume cycle safety beyond it (cycle detection is documented only for `.claude/rules/` symlinks, a different mechanism). What the *apply phase* may then edit is deliberately narrower than what the audit reads:

- **In-repo instruction files** (an imported `AGENTS.md`, `docs/git-instructions.md`, or similar file that exists to instruct AI tools): Step 5 edits the imported file directly for any verdict landing on an imported block — never duplicating imported content back into `CLAUDE.md`. Step 0's flags extend to these files: the encryption check's same-scope verification applies to any imported file the apply phase would edit, and the CI-dependency grep must include imported filenames (a CI script can grep `AGENTS.md` just as it can `CLAUDE.md`), with a flagged file routing to CHALLENGE exactly as a flagged CLAUDE.md block does.
- **Dual-purpose files** (`@README`, `@package.json`, or any imported file whose primary audience/function is not Claude instruction — human-facing docs, machine-read manifests): **never edited by this skill.** A hygiene verdict against their content would damage the file's primary function. Instead, any finding lands on the **import line itself** in CLAUDE.md — which is where the context cost actually lives — e.g. CHALLENGE: *"this imports the full README/package.json into every session; intended, or would a slimmer dedicated instruction file serve better?"* When it's unclear which kind a file is, treat it as dual-purpose and CHALLENGE — never guess an edit.
- **Out-of-repo files** (`@~/...`, absolute paths outside the repo) — **audited but never auto-edited (user decision, 2026-07-08).** These are also exactly the imports the approval gate can have silently disabled (part c). Verdicts on their content surface as CHALLENGEs, each confirmed individually per item 2's out-of-repo discipline; an edit the user approves is applied to the named file with the exact content shown, never bundled.

**Part c — don't misdetect a mention as an import, and don't assume every *external* import is live.** Two correctness requirements, both surfaced by the corrected research: (1) exclude `@path` occurrences inside backticks or fenced code blocks — those are literal mentions, not imports (and treat detection as best-effort against the documented parsing rules; the real parser has had edge cases beyond them, per changelog 2.0.67). This requirement is shared by parts a and b — it ships with whichever lands first. (2) An **external** import (one pointing outside the project) can be syntactically present yet permanently inert if the user declined the one-time approval dialog; in-project imports are not gated and need no such hedge. The skill can't detect the declined state from file content — any finding that leans on externally-imported content should hedge accordingly (e.g., in a CHALLENGE, note "if this external import was declined, this content may not be active in-session"), and the SKILL.md note may mention the optional `InstructionsLoaded`-hook companion pattern for users who want ground truth about what loads. This requirement is needed in full by part b; part a needs only its awareness (an in-repo AGENTS.md import is exempt by construction).

**Files:** `skills/claudemd-tidy/SKILL.md` (Step 2 target-finding + report-mode section for part a; Step 1 note + Step 2b import-following for part b, including the global-file's-own-imports case from gap (b); Step 3/Step 4/Step 5 — verdicts on imported blocks, repo-level CHALLENGE semantics, the three-tier edit policy).
**Bump:** MINOR per the suite's table (new detection + import-following capability; no workflow-contract change — same phases, same stop point, and the file-layout clause concerns *this suite's* layout, not files a run touches in a target repo).

### 5. Axis 5 — survey `.claude/rules/` and add it as a RELOCATE destination

**Proposed fix (two parts).**

**Part a — read side.** Extend Step 2's survey: list the project's `.claude/rules/*.md` files (recursively), noting for each whether it has `paths:` frontmatter. Unconditional rules are always-loaded context with CLAUDE.md-equal priority, so Step 2b's Consistent? and Redundant-by-order comparisons must include their content — a CLAUDE.md line contradicting or duplicating a rules file is exactly what those tests exist to catch. Path-scoped rules are surveyed for the repo picture (what conventions exist, where) but are not "earlier-loaded" in the Redundant-by-order sense — they load conditionally. Overlap between a CLAUDE.md line and a rules file splits by **load-scope equivalence**, not just evidence strength: a verbatim duplicate in an **unconditional** rule is a normal DELETE with the rule file as the cited location (both copies are always-loaded, so removing one changes nothing semantically); a verbatim duplicate in a **path-scoped** rule routes to **CHALLENGE** — deleting the CLAUDE.md copy would narrow the content from always-loaded to conditionally-loaded, which is a scope decision only the user can make (the CHALLENGE presents exactly that: "keep always-loaded here, or let the path-scoped rule be its only home?"); same-intent-different-wording overlap routes to CHALLENGE in both cases (possible drift, user decides which wording and which home wins). `~/.claude/rules/` (user-level, loaded before project rules) joins the global CLAUDE.md in Step 1's earlier-loaded-scope reading for Redundant-by-order — as does a managed-policy CLAUDE.md when one exists on the machine (axis 2's table; this is the clause that makes axis 2's "Redundant-by-order should account for it" actually happen, rather than staying an unimplemented intention).

**Part b — write side.** Add `.claude/rules/<topic>.md` to Step 2 bullet 4's relocation-home inventory and to the RELOCATE test in the Step 3 verdict table (same row item 1 touches — land together if both ship): content that is a fact or constraint applying only to part of the codebase relocates to a **path-scoped rule** (with a `paths:` glob covering that part); a multi-step *procedure* still relocates to a skill; an always-relevant block stays in CLAUDE.md (or an unconditional rule file when the user already organizes rules that way — follow the repo's existing convention, don't introduce `.claude/rules/` into a repo that doesn't use it without raising it as a question first).

Whether rules files themselves become full line-interrogation *targets* (walked line-by-line like a CLAUDE.md) is deliberately left open below — this item only makes the skill *see* them and *relocate into* them.

**Files:** `skills/claudemd-tidy/SKILL.md` (Step 1 earlier-loaded scopes, Step 2 survey + relocation inventory, Step 2b Consistent?/Redundant-by-order scope note, Step 3 RELOCATE row).
**Bump:** MINOR per the suite's table (new survey scope + new destination = new capability; no workflow-contract change).

## What this plan does not decide

- **Naming**: "Correctly-scoped?", "Rule-vs-memory?" are working names, not final — Step 2b's existing five questions (True/Live/Consistent/Actionable/Redundant-by-order) all read as single adjectives or a compact compound; these are a bit longer. Fine to bikeshed at implementation time, not a blocker.
- Whether item 1 is worth doing at all, given it's flagged as already-substantially-covered — included for completeness since the research surfaced it, not because it's clearly worth a release on its own. (If item 5 ships, item 1 rides along in the same RELOCATE-row edit for free.)
- **Item 4's exact CHALLENGE-escalation threshold** ("looks like unintentional drift") isn't fully specified — "overlapping rule text" needs a concrete evidence rule (grep-based? both files naming the same tool/command? semantic similarity?) analogous to how DELETE's evidence rule was made concrete (cited duplicate location or grep-confirmed dead reference). Resolve at implementation time, don't hand-wave it into the shipped skill.
- **Whether `/memory`'s output actually lists arbitrary `@`-imported files** (a `README`, a `package.json`) the same way it lists CLAUDE.md/CLAUDE.local.md/rules files. Current doc wording names only the latter three; changelog 1.0.94 ("direct editing of all imported memory files") suggests imported files were surfaced at least historically — likely yes, but worth the quick empirical check (create a test import, run `/memory`, see what shows) before item 4 part (c)'s hedging language is finalized.
- **Whether the ~150-line hygiene guardrail should be measured against the effective (import-inclusive) size** of a CLAUDE.md, given the official statement that imports "do not reduce context." Natural corollary of item 4 part b's "effective CLAUDE.md" framing; needs the global hygiene rule's owner (the user) to decide, since it changes what the guardrail means — not assumed here.
- **How the skill distinguishes an "instruction file" from a "dual-purpose file"** (item 4 part b) beyond the clear examples — `AGENTS.md` and `docs/git-instructions.md` are clearly instruction files, `README`/`package.json` clearly dual-purpose, but the boundary needs a stated test at implementation time (proposed default: dual-purpose unless the file's evident sole function is instructing AI tools; unclear → CHALLENGE, never guess).
- **Whether rules files become full line-interrogation targets** (walked line-by-line like a CLAUDE.md, with all seven Step 2b questions) or stay survey-only as item 5 proposes. Unconditional rules are CLAUDE.md-equal context, so the case for full treatment exists — but it multiplies the interrogation surface and belongs to a later pass once item 5's survey has run in practice. (The mission plan's Phase 2 target-class registry is the natural place for that later pass.)
- **Whether DELETE's "duplicated verbatim elsewhere" test should require load-scope equivalence.** Item 5's analysis exposed a latent question in the *current* skill: DELETE cites any verbatim duplicate, but a duplicate in a non-loaded or conditionally-loaded location (a README, a path-scoped rule) means deleting the CLAUDE.md copy removes the content from always-loaded context — a semantic change, not pure dedup. Item 5 handles its own case (unconditional rule = DELETE, path-scoped = CHALLENGE); whether the general DELETE test needs the same qualifier is a skill-level fix to evaluate at implementation, with run-record evidence if any exists.
- **Where confirmation ultimately lives.** These items are specced against the current workflow contract (single Step 4 stop point). The mission plan's Phase 3 autonomy rework (reversibility-gated, user decision 2026-07-08) will relocate *when* approval happens for versioned targets — but the requirements defined here survive it unchanged: out-of-repo writes and user-only questions are always confirmed individually, whatever the surrounding flow becomes.
